## 配置详解

### `defineConfig`
- **作用**：tsup 提供的配置函数，提供类型提示和配置验证
- **说明**：
  - 从 `tsup` 包导入的配置辅助函数
  - 提供完整的 TypeScript 类型支持
  - 帮助在编写配置时获得 IDE 自动补全和类型检查
- **好处**：
  - **类型安全**：编译时检查配置错误
  - **智能提示**：IDE 自动补全可用选项
  - **文档支持**：鼠标悬停可查看选项说明

### `entry: ['src/**/*.ts']`
- **作用**：指定打包的入口文件
- **说明**：
  - 使用 glob 模式匹配要打包的文件
  - `src/**/*.ts` 表示 `src` 目录及其所有子目录中的 `.ts` 文件
  - 每个匹配的文件都会作为独立的入口点进行打包
- **匹配规则**：
  - `src/`：src 目录
  - `**`：匹配任意深度的子目录
  - `*.ts`：匹配所有 `.ts` 文件
- **使用场景**：
  ```typescript
  // 单入口
  entry: ['src/index.ts']
  
  // 多入口（当前配置）
  entry: ['src/**/*.ts']
  // 会打包：
  // - src/index.ts -> dist/index.js
  // - src/utils/helper.ts -> dist/utils/helper.js
  // - src/utils/validator.ts -> dist/utils/validator.js
  ```
- **好处**：
  - **多入口打包**：支持按需导入，每个文件独立打包
  - **Tree-shaking 友好**：只导入需要的文件，未使用的代码会被移除
  - **灵活导入**：可以这样导入 `import { helper } from '@justichentai/utils-template/utils/helper'`

### `format: ['esm']`
- **作用**：指定输出格式为 ES 模块（ESM）
- **说明**：
  - `esm` 表示输出 ES 模块格式，使用 `import/export` 语法
  - 支持现代 JavaScript 的模块系统
  - 与 `package.json` 中的 `"type": "module"` 配合使用
- **可选值**：
  - `'esm'`：ES 模块格式（推荐用于现代项目）
  - `'cjs'`：CommonJS 格式（Node.js 传统格式）
  - `'iife'`：立即执行函数表达式（浏览器全局变量）
  - 可以同时指定多个格式：`['esm', 'cjs']`
- **示例**：
  ```typescript
  // 输出 ESM 格式
  export function helper() { }
  
  // 而不是 CommonJS 格式
  // module.exports = { helper() { } }
  ```
- **好处**：
  - **现代标准**：符合 ES 模块规范
  - **Tree-shaking**：打包工具可以更好地进行代码优化
  - **浏览器原生支持**：现代浏览器原生支持 ES 模块

### `minify: true`
- **作用**：启用代码压缩，减小输出文件体积
- **说明**：
  - 压缩 JavaScript 代码，移除空白、注释，缩短变量名等
  - 使用 esbuild 内置的压缩功能，速度极快
  - 生产环境推荐启用
- **压缩效果**：
  ```typescript
  // 压缩前
  export function calculateSum(a: number, b: number): number {
    return a + b;
  }
  
  // 压缩后（示例）
  export function calculateSum(a,b){return a+b}
  ```
- **好处**：
  - **减小体积**：减少文件大小，提升加载速度
  - **提升性能**：更小的文件意味着更快的下载和解析
  - **节省带宽**：减少服务器和用户的带宽消耗
- **注意事项**：
  - 压缩后的代码可读性差，不适合调试
  - 开发环境通常关闭压缩以便调试

### `clean: true`
- **作用**：在构建前清理输出目录
- **说明**：
  - 每次构建前自动删除输出目录（通常是 `dist`）中的旧文件
  - 确保输出目录只包含最新构建的文件
  - 避免旧文件残留导致的问题
- **工作原理**：
  ```typescript
  // 构建流程
  1. 清理 dist 目录（删除所有旧文件）
  2. 编译 TypeScript 代码
  3. 生成新的输出文件到 dist 目录
  ```
- **好处**：
  - **避免残留**：防止旧文件干扰新构建
  - **保持干净**：输出目录始终是最新的构建结果
  - **减少错误**：避免引用已删除但仍在目录中的文件

### `dts: true`
- **作用**：生成 TypeScript 类型定义文件（`.d.ts`）
- **说明**：
  - 自动为每个入口文件生成对应的 `.d.ts` 类型定义文件
  - 这些文件提供 TypeScript 类型信息，供其他项目使用
  - 与 `package.json` 中的 `types` 字段配合使用
- **生成示例**：
  ```typescript
  // src/index.ts
  export function helper(): void { }
  
  // 自动生成 dist/index.d.ts
  export declare function helper(): void;
  ```
- **好处**：
  - **类型支持**：使用该包的项目可以获得完整的 TypeScript 类型提示
  - **自动生成**：无需手动维护类型定义文件
  - **提升体验**：IDE 自动补全、类型检查等功能正常工作
- **注意事项**：
  - 需要安装 `typescript` 作为开发依赖
  - 类型定义文件会与 JavaScript 文件一起输出到 `dist` 目录

### `target: 'es5'`
- **作用**：指定编译目标为 ES5，确保兼容性
- **说明**：
  - 将现代 JavaScript 语法转换为 ES5 语法
  - ES5 是广泛支持的 JavaScript 版本，兼容性最好
  - 确保代码可以在旧版浏览器和 Node.js 环境中运行
- **转换示例**：
  ```typescript
  // 源代码（ES6+）
  const arrow = () => { }
  class MyClass { }
  
  // 编译为 ES5
  var arrow = function() { }
  function MyClass() { }
  ```
- **可选值**：
  - `'es5'`：ES5 语法（最佳兼容性）
  - `'es2015'` / `'es6'`：ES2015 语法
  - `'es2017'`：ES2017 语法
  - `'es2020'`：ES2020 语法（现代浏览器）
  - `'esnext'`：最新语法（需要运行时支持）
- **权衡**：
  - **ES5**：兼容性最好，但文件体积较大（需要更多转换代码）
  - **ES2020+**：文件体积小，但需要现代环境支持
  - 根据目标用户选择合适的目标版本

### `external: ['@swc/helpers']`
- **作用**：将指定包标记为外部依赖，不打包进输出文件
- **说明**：
  - `@swc/helpers` 不会被打包进输出文件
  - 运行时从 `node_modules` 中加载该依赖
  - 与 SWC 的 `externalHelpers: true` 配置配合使用
- **工作原理**：
  ```typescript
  // 源代码
  import { _classCallCheck } from '@swc/helpers'
  
  // 打包后（external 配置）
  import { _classCallCheck } from '@swc/helpers'
  // @swc/helpers 不会被内联，保持 import 语句
  
  // 如果不设置 external
  // @swc/helpers 的代码会被打包进输出文件
  ```
- **好处**：
  - **减小体积**：不重复打包外部依赖
  - **共享代码**：多个包可以共享同一份 `@swc/helpers` 代码
  - **便于更新**：更新依赖时只需更新 `node_modules`，无需重新打包
- **使用场景**：
  - 大型依赖库（如 React、Vue）
  - 共享的辅助函数库（如 `@swc/helpers`）
  - Node.js 内置模块（如 `fs`、`path`）

### `sourcemap: true`
- **作用**：生成 source map 文件，便于调试
- **说明**：
  - 生成 `.map` 文件，映射编译后的代码到原始源代码
  - 调试时可以查看原始 TypeScript 代码而不是编译后的 JavaScript
  - 生产环境通常关闭以减小文件体积
- **生成文件**：
  ```typescript
  // 输出文件
  dist/index.js
  dist/index.js.map  // source map 文件
  ```
- **好处**：
  - **便于调试**：在浏览器调试工具中查看原始 TypeScript 代码
  - **错误定位**：错误堆栈指向原始源代码位置
  - **开发体验**：提升开发时的调试效率
- **工作原理**：
  ```typescript
  // 原始代码 (src/index.ts:5)
  export function helper() {
    console.log('Hello')
  }
  
  // 编译后 (dist/index.js:1)
  export function helper(){console.log("Hello")}
  
  // source map 将 dist/index.js:1 映射回 src/index.ts:5
  ```
- **注意事项**：
  - source map 文件会增加构建时间
  - 生产环境可以考虑关闭或使用 `hidden` source map（不暴露给浏览器）
